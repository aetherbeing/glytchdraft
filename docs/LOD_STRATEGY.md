# LOD_STRATEGY

How GlytchDraft / Miami Slice stays rich at metropolitan scale without
melting the viewer's machine. The strategy applies to Blender now and
to Unity / Unreal / VR / AR later.

For per-scene Blender conventions, see `BLENDER_IMPORT_NOTES.md`. For
the pipeline that produces the assets named here, see
`HERO_TILE_PIPELINE.md` and `MASSING_FROM_LIDAR.md`.

---

## The three-layer model

Every part of the city is represented at **three layers**. These are
not LODs of the same thing — they are different representational
registers that complement each other.

```
┌─────────────────────────────────────────────────────────────┐
│  REALITY  /  EVIDENCE                                       │
│  ▶ LiDAR point clouds (PLY)                                 │
│  ▶ preserves what the city actually is: rooflines, edges,   │
│    facade roughness, vegetation, water, weird true geometry │
│  ▶ source of TRUTH                                          │
├─────────────────────────────────────────────────────────────┤
│  INTERPRETATION  /  MASSING                                 │
│  ▶ extruded prisms from footprints + LiDAR-derived heights  │
│  ▶ one mesh per building (LOD0), or one rotated-bbox per    │
│    building (LOD1), or merged-block (LOD2)                  │
│  ▶ light, navigable, BIM-adjacent                           │
│  ▶ source of FORM                                           │
├─────────────────────────────────────────────────────────────┤
│  ATMOSPHERE  /  MEANING                                     │
│  ▶ Order overlays, AI companion markers, fog, labels,       │
│    color fields, cinematic cards, symbolic traces           │
│  ▶ very lightweight; emissive, transparent, sparse          │
│  ▶ source of SYMBOL                                         │
└─────────────────────────────────────────────────────────────┘

Evidence → Interpretation → Meaning
LiDAR     → Building Masses → Orders + AI Companions
```

The user sees all three at once. The system loads what the current
view actually needs.

## Point-cloud LODs

Spatial subsamples of the LiDAR, generated by PDAL's `filters.sample`
in EPSG:32617 (so the radius is true meters, not Web-Mercator-stretched
meters). Higher numbers = sparser. Each class is its own family.

| Layer | Class | LOD0 (hero) | LOD1 (interactive) | LOD2 (context) |
|---|---|---|---|---|
| buildings | ASPRS 6 | 0.25 m  (4.9M pts) | 0.5 m  (1.7M pts) | 1.0 m  (556K pts) |
| ground    | ASPRS 2 | 1.0 m  (2.3M pts) | 2.0 m  (720K pts) | — |
| water     | ASPRS 9 | 1.0 m  (2.2M pts) | 2.0 m  (670K pts) | — |

Files live at:
```
data_processed/miami/hero_tile/pointcloud/
  hero_tile_building_32617_0p25m.ply   (LOD0)
  hero_tile_building_32617_0p5m.ply    (LOD1)
  hero_tile_building_32617_1m.ply      (LOD2)
  hero_tile_ground_32617_1m.ply        (LOD0)
  hero_tile_ground_32617_2m.ply        (LOD1)
  hero_tile_water_32617_1m.ply         (LOD0)
  hero_tile_water_32617_2m.ply         (LOD1)
```

### Why these spacings

- **Buildings 0.25 m** — facades and openings remain visually legible.
  This is the cinematic / inspection tier.
- **Buildings 0.5 m** — silhouettes still read at street-camera
  distances. Edges hold; small details disappear. Default interactive
  tier.
- **Buildings 1.0 m** — massing tier. Useful as a "ghost" point cloud
  paired with the building masses for a hybrid look.
- **Ground 1.0 m / 2.0 m** — terrain doesn't reward density. 1 m
  reads as a continuous surface; 2 m reads as a textured plane.
- **Water 1.0 m / 2.0 m** — water is essentially flat. 2 m is fine
  for far context; 1 m only when waves or wakes are visible.

### What never gets simplified

- The **raw LAZ** in `OneDrive/Desktop/GLYTCHDRAFT_MIAMI/`. Every PLY
  is a derivative. Re-extract at finer spacing any time.

## Massing LODs

See `MASSING_FROM_LIDAR.md` for the algorithm. The outputs at
`data_processed/miami/hero_tile/blender_ready/masses/` are:

| LOD | File | Geometry per building | When to use |
|---|---|---|---|
| LOD0 | `hero_tile_building_masses_LOD0_individual.obj` | exact footprint polygon × p90 height | mid-distance navigation; per-building selection |
| LOD1 | `hero_tile_building_masses_LOD1_simplified.obj` | minimum-rotated-rectangle × p90 height | far context; merged-block precursor |
| (LOD2) | reserved | merged adjacent footprints | very far / skyline cards |

Each mass carries a `UNIQUEID` matching the source footprint, so the
AI Companion layer can address it by name.

A sibling **`hero_tile_building_masses_metadata.geojson`** carries
the same 2,819 polygons in 2D plus the computed height stats
(`height_p50`, `height_p90`, `height_max`, `ground_z`,
`estimated_height`, `point_count_inside`, `source_quality`,
`UNIQUEID`). This is what agents read; the OBJs are for the viewport.

## Atmosphere / Meaning layer

Lightweight by definition:

- **Order overlays** — empties or thin transparent slabs (e.g.
  `order_mirrorsweat_field`, `order_pink_opaque_field` in
  `06_order_overlays`). No geometry weight; a couple of materials.
- **AI companion markers** — empties in `07_ai_companion_markers`.
  At runtime they may bloom into UI elements, but in the .blend
  they're just named points in space.
- **Fog / haze** — Blender world / volume shaders. No per-object
  cost.
- **Labels / cards** — text or image planes; created on demand.

## Default visibility by viewing distance

This is the contract between the scene's collection structure and the
runtime camera. The Blender scene (`miami_hero_tile_v001.blend`) ships
with the default-visible LODs preset; the others are loaded into the
file but `exclude`d from the active view layer so they don't render.

| Camera distance | Buildings | Ground | Water | Masses |
|---|---|---|---|---|
| close (< 100 m) | LOD0 0.25 m | LOD0 1 m | LOD0 1 m | hide (use points) |
| mid (100–1000 m) | LOD1 0.5 m | LOD0 1 m | LOD0 1 m | LOD0 individual |
| far (1000–3000 m) | LOD2 1 m or hide | LOD1 2 m | LOD1 2 m | LOD0 or LOD1 |
| skyline (> 3000 m) | hide | LOD1 2 m | LOD1 2 m | LOD1 simplified |

In Blender, the live toggle is a per-collection `exclude` flag (the
checkbox next to each collection in the Outliner). In Unity/Unreal,
this becomes an LOD group on each prefab. The naming convention is
identical across both, so a tool can swap LOD0 for LOD2 by string
match.

## Modes (user-facing presets)

These are not collections — they're presets that flip which
collections are excluded. Names are stable; the AI layer (specifically
the Cinematic Director) can request a mode by name.

| Mode | What's visible | Why |
|---|---|---|
| `navigation` (default) | masses LOD0 + buildings LOD2 + ground LOD0 + footprints | smooth pan/zoom, recognizable city |
| `inspection` | + buildings LOD0 + ground LOD0 | dense detail at a chosen district |
| `cinematic` | + buildings LOD0 + masses LOD0 + Order overlays + lights | slower, beautiful for renders |
| `architectural` | masses LOD0 + footprints, NO point clouds | clean BIM-style read |
| `evidence_only` | LiDAR LOD0s only, no masses | raw spatial truth, no interpretation |
| `meaning_only` | masses + Orders + companion markers, no LiDAR | symbolic / map view |

The mode is a runtime concept; the .blend just supplies the
collections. Each mode is implemented in code as a list of
collection names to enable.

## Performance budget (per platform)

Rough targets — adjusted with real testing.

| Platform | Max simultaneous points | Max simultaneous mass faces | Notes |
|---|---|---|---|
| Blender desktop (Eevee) | ~10M | ~5M tris | works for the current hero tile |
| Unity URP (desktop) | ~5M (point primitives) | ~3M tris | bake LODs into prefabs |
| Unreal (Nanite) | n/a (Nanite handles polys) | ~10M tris | point-cloud needs Niagara or custom |
| VR (Quest 3 standalone) | ~500K | ~1M tris | aggressive LOD2 + masses only |
| Web (three.js, GPU mid) | ~1M | ~500K tris | rely on imposter cards far out |
| AR (mobile) | ~200K | ~200K tris | masses + Order overlays only |

The hero tile at default visibility (buildings LOD1 + ground LOD0 +
water LOD0 + masses LOD0 + footprints) is roughly **4.2M points +
~30K triangles**. Below Blender's threshold; above VR standalone's.

## How to lighten on demand

If a scene exceeds budget, three levers in order:

1. **Bump up by one LOD** — buildings LOD1 → LOD2; ground LOD0 →
   LOD1. Halves to quarters the point count.
2. **Drop a layer** — buildings points entirely (keep masses);
   water points entirely (keep a single plane).
3. **Switch mass LOD0 → LOD1** — simplified rectangles are ~5×
   cheaper than exact polygon prisms.

If still over budget after those three: this scene is bigger than
the platform. Crop the tile.

---

## How to enrich on demand

The opposite move. The hero LAZ has 153.7M points (we currently use
~9.5M after LOD0 extraction). If the user zooms into a hero building
at 0.05 m and the 0.25 m PLY looks too sparse:

```cmd
scripts\hero_tile\_run.bat 02 building 0.05
```

That produces a `hero_tile_building_32617_0p05m.ply` at 24× finer
density. Import it as a temporary LOD-1- (negative-one) tier.
Don't commit it to the LOD ladder by default — it's a per-shot
asset.

## Future: Unity / Unreal handoff

The directory layout maps cleanly to game-engine LOD groups:

- Each `*_LOD0/1/2*.ply` becomes a `LODGroup` entry in Unity, or an
  HLOD level in Unreal.
- The mass OBJs map to Nanite-friendly statics (Unreal) or LOD0/1
  meshes (Unity).
- The collection name discipline (`buildings_LOD0_0p25m`, etc.)
  survives the import — both engines preserve container names.

A future export script under `scripts/hero_tile/06_export_unity.py` /
`06_export_unreal.py` would simply mirror these naming conventions.
